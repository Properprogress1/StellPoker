# ADR-002: UltraHonk as the Proving System

**Status**: Accepted

---

## Context

Stellar Poker needs a ZK proving system that satisfies all of the following:

1. **On-chain verifiable on Soroban** — the verifier must run within Soroban's instruction budget.
2. **Compatible with Noir** — the game logic is written in Noir; the proving system must be Noir's native backend.
3. **Practical proof generation time** — the showdown circuit has ~237 000 backend gates. Proving must complete in under 60 seconds on commodity hardware to be usable.
4. **Constant or small proof size** — proof bytes are stored on-chain as part of each transaction; bloated proofs increase Soroban resource costs.
5. **Supported by coNoir for MPC** — TACEO coNoir must support the proving system for collaborative proof generation.

---

## Decision

Use **UltraHonk** (Barretenberg, BN254 scalar field) as the proving system.

---

## Options Considered

### Option A: Groth16

A highly mature, production-battle-tested SNARK with constant proof size (~200 bytes) and fast on-chain verification.

**Pros:**
- Very small proof (192 bytes).
- On-chain verification is cheap: 3 pairing operations on BN254.
- Widely supported across EVM chains.

**Cons:**
- Requires a per-circuit trusted setup ceremony. Changing a circuit requires a new ceremony.
- Noir's Barretenberg backend does not support Groth16 in the 1.x series — it would require a different toolchain.
- coNoir does not implement Groth16 collaborative proving.

### Option B: PLONK (original / TurboPlonk)

An earlier member of the PlonK family with an updatable SRS (no per-circuit ceremony).

**Pros:**
- Universal SRS — no per-circuit ceremony.
- Proof size ~500 bytes.

**Cons:**
- Superseded by UltraPlonk/UltraHonk in Barretenberg; the original PLONK backend is not actively developed in the Noir ecosystem.
- Less efficient than UltraHonk for custom gates and lookup tables.
- Not supported by recent versions of Nargo.

### Option C: STARKs (e.g., RISC Zero, Plonky2)

Hash-based proving systems with no trusted setup and post-quantum security.

**Pros:**
- No trusted setup of any kind.
- Post-quantum secure.
- Parallelizable proof generation.

**Cons:**
- Proof sizes are large (50–500 KB depending on circuit and hash function). On-chain storage would dominate transaction costs.
- Not natively supported by Noir (would require transpiling ACIR to a different IR).
- Verification on Soroban would require a hash-based verifier contract, which has no native host function acceleration — likely exceeds the instruction budget.

### Option D: UltraHonk (Barretenberg) — **chosen**

The current production backend for Nargo 1.x. Supports custom gates, lookup tables (Plookup), and has an updatable (universal) SRS.

**Pros:**
- Native Nargo/Barretenberg backend — no toolchain changes needed.
- Universal SRS: no per-circuit trusted setup ceremony. Changing circuits only requires regenerating the verification key.
- Constant proof size: **16 256 bytes** for all three circuits regardless of gate count.
- Supported by TACEO coNoir for collaborative proving (coSNARK).
- Soroban Protocol 25 introduces native BN254 host functions (Poseidon2, EC operations, MSM) that accelerate UltraHonk verification. Without these, the verifier would exceed the instruction budget.

**Cons:**
- Proof size (16 256 bytes) is larger than Groth16 (~192 bytes), increasing on-chain transaction size.
- Verification is more expensive than Groth16 in raw operation count, but Soroban's native BN254 host functions (Protocol 25 & 26) bring it within budget.
- UltraHonk is not yet as battle-tested as Groth16 in production deployments.

---

## Consequences

- The `zk-verifier` Soroban contract embeds a circuit-specific **verification key** for each of the three circuits. Keys are generated by `bb write_vk --scheme ultra_honk`.
- Any circuit change requires regenerating the VK and calling `set_vk` on the contract before new proofs can be verified on-chain.
- Proof generation uses `bb prove --scheme ultra_honk`. In MPC mode, this is replaced by `co-noir prove` which runs the collaborative prover.
- The BN254 SRS (Common Reference String) must be downloaded once per machine: `./scripts/download-crs.sh`. The CRS is universal — the same SRS works for all circuits.
- Proof and public input sizes are fixed constants in the `zk-verifier` contract source (`PROOF_BYTES`, `DEAL_BYTES`, `REVEAL_BYTES`, `SHOWDOWN_BYTES`). If Barretenberg changes proof format, these must be updated together.
